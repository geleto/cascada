# Cascada Renaming Audit: Hard Rename to Channels

## Scope

This document defines a hard rename policy for code and code comments.

- In scope: `src/**`, tests, and inline comments/error strings generated by code.
- Out of scope for this pass: general user-facing documentation rewrites.
- Exception: this file is updated as the implementation contract.

## Non-Negotiable Rules

1. No legacy architecture names remain in code/comments.
2. `ValueOutput` is renamed to `VarChannel`.
3. Observation is a command and must be named as such.
4. Use `mutation command` and `observation command` terminology.
5. `var` is internally implemented as a channel, but user-facing docs may describe it separately.
6. Do not use `output` terminology for channels; only keep `output` when it means generic rendered/result output (non-channel meaning).
7. No compatibility aliases. This is a full rename, not a deprecation transition.

## Canonical Vocabulary

- `channel`: runtime state lane (readable and writable).
- `channel declaration`: declaration of a channel (`data`, `text`, `var`, `sink`, `sequence`, etc.).
- `mutation command`: command that mutates channel state.
- `observation command`: command that observes channel state (`snapshot`, peek, `isError`, `getError`, etc.).
- `channel queue` / `command queue`: ordering line where both mutation and observation commands wait and execute in sequence.

Important clarification:
- Channel observation is not a side read outside scheduling. It is an `observation command` that queues and waits in line like other commands.

## Required Text Renames (Strings and Comments)

Replace in code comments and user-visible code-generated text where semantically channel-related:

- `output handler` -> `channel`
- `handler lane` -> `channel lane` or `channel queue`
- `output command` -> `mutation command` or `observation command` (choose by behavior)
- `output observation` -> `observation command`
- `value output` / `var-output` / `non-value output` -> `var channel` / `variable` / `declared channel` (context-specific)
- `unsupported output type` -> `unsupported channel type`
- `Output 'X' ...` -> `Channel 'X' ...` (where referring to channel declarations)

Do not rename generic render-result usage such as:
- "template output" meaning final rendered text,
- "script output" meaning function return/result,
unless the text actually refers to channel entities.

## Command Taxonomy Requirements

Every command type must be classifiable as one of:

- `mutation command`
- `observation command`

Naming should reflect this when practical. Examples:
- `SnapshotCommand` -> observation command category.
- state-write commands -> mutation command category.

Queue semantics must be reflected in comments:
- observation commands are queued and ordered the same way as mutation commands.

## File/Module Naming Requirements

Channel-related files/modules should not keep legacy channel-meaning `output` names.

Examples:
- `runtime/output.js` should become channel-named.
- any channel-specific `output-*` module names should be channel-named.

If a file name includes `output` but is strictly about generic final render output, it may remain.

## Enforcement Checklist

- No channel-meaning `Output*` symbols remain.
- No channel-meaning `declareOutput/getOutput` remains.
- No channel-meaning `handler` terminology remains in identifiers/comments.
- No `ValueOutput` symbol remains; only `VarChannel`.
- Observation paths are named and documented as commands.
- Tests updated for renamed symbols and strings.
